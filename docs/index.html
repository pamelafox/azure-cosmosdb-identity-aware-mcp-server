Before MCP

Before MCP, connecting to a database required one approach, Slack required another, GitHub a third. Every time a new application wanted access to the same services, someone had to wire them all up again.

Model Context Protocol

An open protocol that defines how AI applications get context from external tools and data sources. It was created by Anthropic and is now part of the Agentic AI Foundation under the Linux Foundation.

MCP defines a standard way for models to get context. Covered in depth in session 1. The protocol homepage is modelcontextprotocol.io.

MCP architecture

MCP hosts like VS Code contain MCP clients that connect to servers. Servers expose tools (functions the LLM can call), resources (read-only data), and prompts (instruction templates).

Anatomy of our MCP server

Components of our MCP server

This identity-aware MCP server combines four pieces:

FastMCP

Python framework for the MCP server implementation.

Azure Cosmos DB

Stores per-user expense data with partitioning by user ID.

Microsoft Entra ID

Handles user authentication and issues the OAuth access token.

Microsoft Graph

Checks directory data after sign-in, including admin group membership.

This server authenticates users via Entra, stores their expenses in Cosmos DB partitioned by user ID, and checks group membership via Graph API for admin-only tools.

Server architecture

VS Code authenticates directly with Entra, gets a token, and sends it to the FastMCP server. The server validates the JWT and queries Cosmos DB with the user's OID. For admin checks, the server uses OBO flow to call Graph API.

Covered in session 3. Private network was session 2, key-based and OAuth were session 3. For this demo, we need OAuth because we need to know WHO the user is to store and retrieve their personal data.

OAuth 2.1 flow for MCP

  1. MCP client sends request to server → 401 Unauthorized
  2. Client discovers auth server via Protected Resource Metadata (PRM)
  3. Client redirects user to authorization server (Entra login)
  4. User authenticates and consents
  5. Authorization server issues access token
  6. Client sends requests with Authorization: Bearer <token>
  7. Server validates JWT and extracts user identity

MCP auth specification

The key flow: client hits the server, gets a 401, discovers the auth server through PRM, authenticates with Entra, gets a token, and then sends authenticated requests. The server never needs to proxy OAuth — it just validates JWTs.

Our approach: RemoteAuthProvider

The server just validates JWTs — no OAuth proxy needed.

  • Server serves PRM endpoint pointing client directly to Entra
  • VS Code authenticates directly with Entra (pre-registered client)
  • Server validates tokens using Entra's public signing keys
  • No client secret needed on the server for auth

Server logs show the simplicity — no /authorize, /token, or /consent endpoints:


POST /mcp HTTP/1.1  → 401 Unauthorized
GET  /.well-known/oauth-protected-resource/mcp  → 200 OK
POST /mcp HTTP/1.1  → 200 OK  (with Bearer token)
This is Option 3 from the plan. VS Code is pre-registered as an authorized application on our Entra app registration. When the PRM points to Entra, VS Code uses its own first-party Entra GUID and authenticates via the macOS/Windows authentication broker. The server just validates the resulting JWT.

Setting up RemoteAuthProvider


from fastmcp import FastMCP
from fastmcp.server.auth import RemoteAuthProvider
from fastmcp.server.auth.providers.azure import AzureJWTVerifier

verifier = AzureJWTVerifier(
    client_id=entra_client_id,
    tenant_id=os.environ["AZURE_TENANT_ID"],
    required_scopes=["user_impersonation"],
)

auth = RemoteAuthProvider(
    token_verifier=verifier,
    authorization_servers=[
        f"https://login.microsoftonline.com/{os.environ['AZURE_TENANT_ID']}/v2.0"
    ],
    base_url=base_url,
)

mcp = FastMCP("Expenses Tracker", auth=auth)
AzureJWTVerifier validates Entra v2.0 tokens using the public signing keys. RemoteAuthProvider just serves the PRM endpoint, pointing clients directly to Entra. No proxy endpoints, no client secret for MCP auth. Same code works locally and in production.

Extracting user identity from the token


from fastmcp.server.dependencies import get_access_token
from fastmcp.server.middleware import Middleware, MiddlewareContext

class UserAuthMiddleware(Middleware):
    def _get_user_id(self):
        token = get_access_token()
        if not (token and hasattr(token, "claims")):
            return None
        return token.claims.get("oid")  # Entra Object ID

    async def on_call_tool(self, context: MiddlewareContext, call_next):
        user_id = self._get_user_id()
        if context.fastmcp_context is not None:
            await context.fastmcp_context.set_state("user_id", user_id)
        return await call_next(context)

mcp = FastMCP("Expenses Tracker", auth=auth,
              middleware=[UserAuthMiddleware()])
Middleware extracts the user's Object ID (oid) from the JWT claims on every tool call. This OID uniquely identifies the user in Entra and becomes the key for storing their data in Cosmos DB. The state is passed through the FastMCP context so tools can access it.

Key requirements: requestedAccessTokenVersion: 2, identifierUris,
and VS Code's GUID in preAuthorizedApplications

The app registration exposes a user_impersonation scope and pre-authorizes VS Code's Entra GUID. The identifierUris must be set for preAuthorizedApplications to work. requestedAccessTokenVersion: 2 is required because AzureJWTVerifier only supports v2 tokens. Without these settings, auth silently fails.

Integrating Cosmos DB
with MCP servers

Why Cosmos DB?

  • Serverless or provisioned — scale to zero or handle millions of requests
  • Low-latency reads and writes — single-digit millisecond response times
  • Partition key design — natural per-user data isolation
  • Identity-based access — managed identity, no connection strings
  • Python SDK — async support with azure-cosmos
Cosmos DB is a great fit for MCP servers because each user's data can be isolated by partition key. Managed identity means no secrets to manage. The Python SDK supports async operations which fits well with FastMCP's async tools.

Partition key design for per-user data

Each expense document includes a user_id field set to the Entra Object ID:


{
  "id": "a1b2c3d4-...",
  "user_id": "00000000-0000-...",
  "date": "2026-03-31",
  "amount": 20.00,
  "category": "food",
  "description": "Avocado toast",
  "payment_method": "visa"
}

Partition key = /user_id

  • All of a user's expenses are in the same partition
  • Queries with WHERE user_id = @uid are single-partition (fast)
  • Users cannot read other users' partitions
  • Cross-partition queries only needed for admin stats
The partition key is the user_id from the Entra OID. This means all of a user's expenses live in the same logical partition. Queries filtered by user_id are single-partition and very efficient. Cross-partition queries are only used for admin-level aggregate stats.

Setting up the Cosmos DB client


from azure.cosmos.aio import CosmosClient
from azure.identity.aio import ManagedIdentityCredential, AzureDeveloperCliCredential

# Production: managed identity. Local: azd CLI credential.
if RUNNING_IN_PRODUCTION:
    azure_credential = ManagedIdentityCredential(
        client_id=os.environ["AZURE_CLIENT_ID"])
else:
    azure_credential = AzureDeveloperCliCredential(
        tenant_id=os.environ["AZURE_TENANT_ID"])

cosmos_client = CosmosClient(
    url=f"https://{os.environ['AZURE_COSMOSDB_ACCOUNT']}.documents.azure.com:443/",
    credential=azure_credential,
)
cosmos_db = cosmos_client.get_database_client(os.environ["AZURE_COSMOSDB_DATABASE"])
cosmos_container = cosmos_db.get_container_client(
    os.environ["AZURE_COSMOSDB_USER_CONTAINER"])

No connection strings — identity-based access everywhere.

The Cosmos DB client uses managed identity in production and the azd CLI credential for local development. No connection strings or secrets. The container is configured with user_id as the partition key.

Tool: add_user_expense


@mcp.tool
async def add_user_expense(
    date: Annotated[date, "Date of the expense in YYYY-MM-DD format"],
    amount: Annotated[float, "Positive numeric amount of the expense"],
    category: Annotated[Category, "Category label"],
    description: Annotated[str, "Human-readable description of the expense"],
    payment_method: Annotated[PaymentMethod, "Payment method used"],
    ctx: Context,
):
    """Add a new expense to Cosmos DB."""
    user_id = await ctx.get_state("user_id")
    if not user_id:
        return "Error: Authentication required"

    expense_item = {
        "id": str(uuid.uuid4()),
        "user_id": user_id,
        "date": date.isoformat(),
        "amount": amount,
        "category": category.value,
        "description": description,
        "payment_method": payment_method.value,
    }
    await cosmos_container.create_item(body=expense_item)
    return f"Successfully added expense: ${amount} for {description}"
The tool uses typed Annotated parameters so the LLM knows exactly what to fill in. The user_id comes from middleware (extracted from the JWT). The expense document includes the user_id as both a field and the partition key value.

Tool: get_user_expenses


@mcp.tool
async def get_user_expenses(ctx: Context):
    """Get the authenticated user's expense data from Cosmos DB."""
    user_id = await ctx.get_state("user_id")
    if not user_id:
        return "Error: Authentication required"

    query = "SELECT * FROM c WHERE c.user_id = @uid ORDER BY c.date DESC"
    parameters = [{"name": "@uid", "value": user_id}]
    expenses_data = []

    async for item in cosmos_container.query_items(
        query=query, parameters=parameters, partition_key=user_id
    ):
        expenses_data.append(item)

    return json.dumps([{
        "date": e.get("date"), "amount": e.get("amount"),
        "category": e.get("category"), "description": e.get("description"),
        "payment_method": e.get("payment_method"),
    } for e in expenses_data], indent=2)

Passing partition_key=user_id makes this a single-partition query.

The query filters by user_id and passes the partition key to the SDK, making it a single-partition read. Parameterized queries prevent injection. The result is JSON that the LLM can summarize or analyze.

Role-based access with
Microsoft Graph

Why role-based access?

Some tools should only be available to certain users:

  • Regular users: add and view their own expenses
  • Admins: view aggregate stats across all users

Use Entra security groups to define roles:

  • Create an "MCP Admins" group in Entra
  • Add admin users to the group
  • Server checks group membership via Microsoft Graph API
Not all tools should be available to all users. Regular users can only manage their own expenses. Admins can see stats across all users. We use Entra security groups to define who is an admin.

On-Behalf-Of (OBO) flow for Graph API

  1. User sends MCP request with their Entra access token
  2. Server exchanges that token for a Graph API token via OBO flow
  3. Server calls Graph: /me/transitiveMemberOf?$filter=id eq '{group_id}'
  4. If user is in the admin group → tool executes
  5. If not → tool is hidden and blocked

Production: OBO uses managed identity as federated credential (no secret).
Local dev: OBO uses a client secret from the dev app registration.

The OBO flow lets the server act on behalf of the user to call Graph API. In production, the managed identity provides the federated credential so no client secret is needed. Locally, a separate dev app registration with a client secret is used.

Checking group membership via Graph API


async def check_user_in_group(graph_token: str, group_id: str) -> bool:
    """Check if the user is a member of the specified group."""
    async with httpx.AsyncClient() as client:
        url = (
            "https://graph.microsoft.com/v1.0/me/transitiveMemberOf"
            "/microsoft.graph.group"
            f"?$filter=id eq '{group_id}'&$count=true"
        )
        response = await client.get(url, headers={
            "Authorization": f"Bearer {graph_token}",
            "ConsistencyLevel": "eventual",
        })
        response.raise_for_status()
        return response.json().get("@odata.count", 0) > 0
The Graph API call checks transitive membership, so nested groups work too. The ConsistencyLevel: eventual header is required for $count queries.

Component-level authorization in FastMCP


async def require_admin_group(ctx: AuthContext) -> bool:
    """Auth check: user must be in the admin group."""
    admin_group_id = os.environ.get("ENTRA_ADMIN_GROUP_ID", "")
    if not admin_group_id or not confidential_client or ctx.token is None:
        return False

    graph_token = confidential_client.acquire_token_on_behalf_of(
        user_assertion=ctx.token.token,
        scopes=["https://graph.microsoft.com/.default"],
    )
    if "error" in graph_token:
        return False

    return await check_user_in_group(
        graph_token["access_token"], admin_group_id)
